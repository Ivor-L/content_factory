# Agent Teams 规划提示词模板库

> 适用于 Claude Code v2.1.32+，需开启 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`

---

## 一、通用启动模板

### 基础团队启动
```
创建一个 Agent Team 来完成以下任务：[任务描述]

团队结构：
- Team Lead（你）：负责协调、分配任务、汇总结果
- Teammate A（[角色名]）：负责 [具体职责]
- Teammate B（[角色名]）：负责 [具体职责]
- Teammate C（[角色名]）：负责 [具体职责]

要求：
1. 各 Teammate 独立并行工作，不要等待彼此
2. 每个 Teammate 完成后向 Lead 汇报结果
3. Lead 汇总所有结果后给我最终报告
4. 所有 Teammate 完成工作后执行 cleanup
```

## 二、场景化提示词模板

### 🔍 场景一：并行代码审查

```
创建一个代码审查 Agent Team，对 [文件路径/PR编号] 进行全面审查。

生成 3 个审查员 Teammate，各自使用不同视角独立审查，然后互相挑战对方的发现：

Teammate 1 - 安全审查员：
专注于安全漏洞：SQL注入、XSS、认证缺陷、敏感数据暴露、权限越权。
审查完成后，将发现的问题按严重程度（Critical/High/Medium/Low）分类列出。

Teammate 2 - 性能分析师：
专注于性能问题：N+1查询、内存泄漏、不必要的重渲染、缺失的索引、慢查询。
评估每个问题的性能影响，给出量化估计（如"可减少 X% 响应时间"）。

Teammate 3 - 测试覆盖审计员：
专注于测试质量：覆盖率缺口、边界条件未测试、Mock 使用不当、测试隔离问题。
列出缺失的测试用例，并给出优先级排序。

协作要求：
- 每个 Teammate 完成初审后，互相传阅对方的发现
- 对有争议的问题进行辩论，达成共识
- Lead 汇总三方意见，生成最终审查报告，包含：优先修复清单 + 改进建议
```

### 🐛 场景二：并行 Bug 调查（竞争假设）

```
用户报告了以下 Bug：[Bug 描述]
重现步骤：[步骤]
环境：[环境信息]

创建 Agent Team，使用"竞争假设"方法并行调查根本原因。

生成 4 个调查员 Teammate，每人持不同假设：

Teammate 1 - 假设：数据层问题
调查方向：数据库查询、事务、数据一致性、缓存失效
查看文件：[数据库相关文件路径]
目标：证明或证伪"数据层导致此 Bug"

Teammate 2 - 假设：状态管理问题
调查方向：前端状态、Redux/Zustand store、组件生命周期、竞态条件
查看文件：[状态管理相关文件路径]
目标：证明或证伪"状态管理导致此 Bug"

Teammate 3 - 假设：API/网络问题
调查方向：API 响应格式、错误处理、超时、CORS、认证 Token
查看文件：[API 相关文件路径]
目标：证明或证伪"API层导致此 Bug"

Teammate 4 - 假设：并发/时序问题
调查方向：异步操作、事件循环、定时器、WebSocket、并发请求
查看文件：[相关文件路径]
目标：证明或证伪"并发问题导致此 Bug"

协作要求：
- 各 Teammate 完成调查后，互相质疑对方的结论
- 通过直接通信进行"科学辩论"
- 最终形成共识：最可能的根本原因 + 修复方案
- Lead 生成包含：根因分析 + 修复代码 + 回归测试的完整报告
```

### 🏗️ 场景三：新功能并行开发

```
需要实现新功能：[功能名称]
需求文档：[需求描述]
技术栈：[技术栈信息]

创建 Agent Team 进行并行开发，每个 Teammate 负责独立模块（无文件冲突）：

生成 3 个开发者 Teammate：

Teammate 1 - 后端 API 开发者：
负责文件：[后端文件路径，如 src/api/, src/services/]
任务：
1. 设计并实现 RESTful API 端点
2. 实现业务逻辑层
3. 编写 API 单元测试
4. 更新 API 文档
要求：实现前先制定方案并等待 Lead 批准（plan approval mode）

Teammate 2 - 前端组件开发者：
负责文件：[前端文件路径，如 src/components/, src/pages/]
任务：
1. 实现 UI 组件
2. 对接 API（使用 Mock 数据先行开发）
3. 编写组件测试
4. 确保响应式设计
要求：与 Teammate 1 对齐 API 接口定义后再开始实现

Teammate 3 - 数据库 & 基础设施：
负责文件：[数据库相关文件，如 prisma/schema.prisma, migrations/]
任务：
1. 设计数据库 Schema
2. 编写 Migration 文件
3. 配置必要的环境变量
4. 更新 Dockerfile/配置文件（如有）

协作要求：
- Teammate 3 先完成 Schema 设计，Teammate 1 参考后实现 API
- Teammate 1 完成 API 接口定义后通知 Teammate 2
- 所有人完成后，Lead 进行集成测试
- 禁止编辑对方负责的文件
```

### 🔬 场景四：技术调研与选型

```
我们需要为项目选择 [技术方案/库/架构]，当前项目使用 [技术栈]。
核心需求：[需求描述]

创建 Agent Team，让每个 Teammate 深度研究一个候选方案：

Teammate 1 - 方案 A 研究员（[方案A名称]）：
研究维度：
- 性能基准（查找官方测评或社区测评）
- 与当前技术栈的兼容性
- 学习曲线和文档质量
- 社区活跃度（GitHub stars、issue 响应时间）
- 生产环境案例
- 许可证类型
输出：优缺点对比表 + 集成难度评估 + 推荐指数（1-10）

Teammate 2 - 方案 B 研究员（[方案B名称]）：
[同上维度]

Teammate 3 - 方案 C 研究员（[方案C名称]）：
[同上维度]

Teammate 4 - 魔鬼代言人：
任务：尝试找出每个方案的致命缺陷，质疑其他 Teammate 的正面评价。
重点关注：已知的坑、未来维护风险、供应商锁定风险、安全历史记录。

协作要求：
- 各研究员完成后互相分享报告
- 魔鬼代言人针对每个方案的优点提出反驳
- Lead 综合所有信息，给出最终推荐方案 + 迁移/集成路径
```

### 🧪 场景五：全面测试覆盖

```
项目当前测试覆盖率不足，需要系统性补充测试。
测试框架：[Jest/Vitest/Playwright 等]
覆盖目标：[目标覆盖率，如 80%]

创建 Agent Team，并行补充不同类型的测试：

Teammate 1 - 单元测试专家：
负责范围：[src/utils/, src/lib/, src/services/]
任务：
- 分析现有函数，找出未覆盖的分支
- 重点覆盖：边界值、异常路径、纯函数逻辑
- 为每个文件至少添加 3 个有意义的测试用例
- 确保 Mock 隔离正确

Teammate 2 - 集成测试专家：
负责范围：[src/api/, API 端点测试]
任务：
- 对每个 API 端点编写集成测试
- 覆盖：正常流程、错误场景、权限验证、数据验证
- 使用测试数据库（不影响生产数据）

Teammate 3 - E2E 测试专家：
负责范围：[关键用户流程]
任务：
- 识别最关键的 3-5 个用户流程
- 使用 Playwright 编写 E2E 测试
- 覆盖：登录、核心业务流程、错误处理

完成标准：
- 所有新增测试必须通过
- 不得修改现有测试
- Lead 最后运行完整测试套件并报告覆盖率变化
```

## 三、高级控制模板

### 需要计划审批的高风险任务

```
[任务描述] - 这是一个高风险操作，涉及 [数据库变更/生产环境/核心模块]。

创建 Agent Team，要求所有 Teammate 在执行前提交计划供审批：

生成 [N] 个 Teammate，并为每个 Teammate 要求 plan approval：
"生成 Teammate 后，要求他们先进入 plan 模式，提交实施方案，
 只有我（Lead）审批通过后才能开始实际操作。
 审批标准：方案必须包含回滚步骤、影响评估、测试验证方法。"

[Teammate 职责分配...]
```

### 指定模型和数量

```
创建 Agent Team 完成 [任务]。

配置要求：
- 生成 4 个 Teammate
- 每个 Teammate 使用 claude-sonnet-4-5 模型
- Lead 继续使用当前模型

[任务详细描述...]
```

### 强制等待模式（防止 Lead 抢先工作）

```
[任务描述]

创建 Agent Team 后：
1. 将所有任务分配给 Teammate
2. **你（Lead）必须等待所有 Teammate 完成后再做任何工作**
3. 不要自己开始实现任何代码
4. 只在以下情况介入：Teammate 遇到阻塞、需要跨 Teammate 协调
5. 所有人完成后，你负责整合和最终验收

[Teammate 分工...]
```

## 四、收尾与质量控制

### 标准收尾提示词

```
所有 Teammate 已完成工作，请执行以下收尾流程：

1. 汇总所有 Teammate 的工作成果
2. 检查是否有遗漏的任务或未解决的依赖
3. 运行完整的测试套件，确认所有测试通过
4. 生成工作总结报告：
   - 完成了什么
   - 做了哪些决策及理由
   - 遗留问题（如有）
   - 建议的后续步骤
5. 执行 cleanup，关闭所有 Teammate 并清理团队资源
```

### 质量门控 Hook 建议

在 `.claude/settings.json` 的 hooks 配置中添加：

```json
{
  "hooks": {
    "TaskCompleted": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "npm test -- --passWithNoTests 2>&1 | tail -5",
            "statusMessage": "验证测试通过..."
          }
        ]
      }
    ]
  }
}
```

## 五、快速参考卡

| 场景 | 推荐 Teammate 数 | 模式 | 注意事项 |
|------|-----------------|------|----------|
| 代码审查 | 3 | 互相挑战 | 确保审查维度不重叠 |
| Bug 调查 | 4-5 | 竞争假设 | 明确每人的假设方向 |
| 功能开发 | 3-4 | 独立模块 | 严格划分文件归属 |
| 技术调研 | 3+魔鬼代言人 | 并行研究 | 统一输出格式 |
| 测试补充 | 3 | 类型分工 | 明确各自的文件范围 |

**Token 消耗提示**：每增加 1 个 Teammate，token 消耗线性增加。
建议生产环境任务从 3 个 Teammate 起步，验证后再扩展。

**文件冲突避免**：在任务分配时明确每个 Teammate 只能编辑哪些文件，
Lead 在分配任务时应在提示词中写明"禁止编辑对方的文件"。
