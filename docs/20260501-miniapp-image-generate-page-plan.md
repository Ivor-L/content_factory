# 20260501 小程序图片生成页（多功能切换）执行计划

## 目标
在 Taro 小程序新增一个“图片生成”页面，顶部提供功能切换，包含 3 个子功能：
1. AI 作图：上传参考图、模型选择、提示词输入。
2. 信息图：模板选择 + 文案输入，支持“找灵感”“粘贴”。
3. 图文卡片：Markdown/文案转卡片排版入口与基础预览。

## 范围
- `digital_human_miniapp/taro/src/pages` 新增页面与样式。
- `digital_human_miniapp/taro/src/app.config.ts` 注册新页面。
- `digital_human_miniapp/taro/src/pages/home/index.tsx` 调整“图片生成”入口跳转。
- 文档索引更新：`docs/README.md`。

## 方案对比（至少 2 种）
### 方案 A：单页面多 Tab（本次采用）
- 做法：一个页面承载 3 个功能分区，顶部切换。
- 优点：交互统一、切换快、与参考图一致、开发成本低。
- 缺点：页面状态较多，后续复杂时需组件化拆分。

### 方案 B：三个独立页面
- 做法：每个功能单独页面，通过中间页或顶部导航跳转。
- 优点：职责清晰、单页复杂度低。
- 缺点：来回跳转成本高，和“顶部切换”的目标不一致。

## 兼容性结论
- Next.js/Prisma/Supabase：本次仅改 Taro 前端页面，不涉及服务端 schema、API contract、数据库读写，兼容性风险低。
- n8n/第三方 API：本次先落前端交互壳层，不新增后端依赖，不影响现有工作流。
- 微信小程序运行时：使用 `Taro.chooseImage`、`Taro.getClipboardData`、`Taro.navigateTo` 等通用能力，兼容当前项目架构。

## 风险与回滚
- 风险 1：新页面样式与现有主题不一致。
  - 缓解：复用现有暗色基调和安全区写法。
- 风险 2：首页入口切换后影响原“图片生成”链路。
  - 缓解：仅替换入口目标，原页面保留可独立访问。
- 回滚策略：
  1. 回退 `home/index.tsx` 的入口路由。
  2. 从 `app.config.ts` 移除新页面注册。
  3. 删除新页面目录（`pages/image-generate`）。

## 不确定点与 POC
- 不确定点：信息图/图文卡片是否立即接入正式生成 API。
- POC 结论：先实现完整交互壳层（输入、模板、粘贴、跳灵感、提交按钮），API 在后续对接；该路径可最快验证用户流程。

## 里程碑
1. 新增计划文档与索引。
2. 新增页面（结构 + 样式 + 交互壳层）。
3. 接入首页入口与“找灵感”跳转。
4. 本地静态校验（lint/typecheck）。

## 验收标准
- 首页“图片生成”可进入新页面。
- 顶部可切换 3 个功能：AI 作图 / 信息图 / 图文卡片。
- 信息图支持“找灵感”跳转爆款图文页，支持“粘贴”读取剪贴板。
- 页面在移动端安全区显示正常，无明显布局错乱。

- 可进一步拆分为 `components`（TabBar、ModelPicker、TemplateGrid）降低页面耦合。
