# 2026-05-02 小程序爆款数据中心（分类配置 + 搜索采集）计划

## 目标
- 小程序「爆款广场」中的 `我的` 分类继续展示当前用户个人采集数据。
- 其余分类由管理后台配置，并支持后台触发小红书搜索采集入库。
- 小程序按后台配置动态展示分类，并按分类读取后台采集的公共爆款数据。

## 范围
- 管理后台：
  - 新增「爆款数据中心」页面，维护分类配置与采集参数。
  - 支持对单个分类触发采集（`GET /api/v1/xiaohongshu/app_v2/search_notes`）。
- 服务端 API：
  - 管理端配置读写与采集接口（仅管理员）。
  - 小程序端读取分类配置接口（公开已发布配置）。
  - 爆款列表查询新增“共享数据范围”能力（用于非 `我的` 分类）。
- 小程序端：
  - 爆款页分类改为动态读取后台配置（保留 `全行业` 与 `我的`）。
  - `我的` 走现有个人数据源；其他分类走共享爆款库。

## 方案对比
1. 方案 A：新增独立表（分类表 + 笔记表）
- 优点：结构强约束，易扩展复杂运营字段。
- 缺点：需要新增迁移与更多后台 CRUD，首版实现成本高。

2. 方案 B：复用现有配置表 + 爆款素材表（推荐）
- 优点：无需新增 schema，改造成本低，上线快。
- 缺点：配置依赖 JSON 结构，需要服务端做归一化校验。

## 兼容性
- Next.js App Router：新增 admin/miniapp API 路由，不影响现有入口。
- Prisma/Supabase：复用 `monetization_square_configs` 与 `viral_reference_items`，首版无需迁移。
- 小程序：仅调整爆款页分类来源与查询参数，详情链路保持兼容。

## 风险与回滚
- 风险：采集接口返回结构波动导致部分字段映射失败。
  - 处理：解析逻辑多路径兜底，最小字段（id/title/cover）缺失时跳过入库。
- 风险：共享数据与用户私有数据混淆。
  - 处理：共享数据使用固定 owner 标识隔离，查询显式区分 scope。
- 回滚：
  - 小程序分类读取失败时回退到本地默认分类。
  - 后台采集可停用，不影响现有 `我的` 分类链路。

## 分阶段里程碑
1. M1：配置模型与接口
- 定义数据中心配置结构与归一化逻辑。
- 完成管理端 GET/PUT 接口与小程序端 GET 配置接口。

2. M2：采集入库
- 完成管理员采集接口，调用 `search_notes` 并写入共享爆款库。

3. M3：前端接入
- 管理后台页面可配置分类并触发采集。
- 小程序爆款页动态分类与分源查询联调。

## 验收标准
- `我的` 分类仅展示当前用户个人采集数据。
- 后台新增/修改分类后，小程序分类可同步展示。
- 对某分类执行采集后，该分类下能看到新增笔记。
- `npm run lint`、`npm run typecheck` 通过。
