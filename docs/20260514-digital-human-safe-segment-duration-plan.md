# 数字人安全分段时长修复计划

## 目标

修复数字人文字驱动分段贴近 32 秒模型硬上限时，实际生成超时并丢失末尾文字的问题。

## 范围

- Web 数字人分段时长估算与提示。
- 小程序数字人分段时长估算与提示。
- 后端提交 n8n 前的自动分段阈值。

## 调研结论

方案 A：保持 32 秒硬上限，只修改提示文案。
- 优点：改动小。
- 风险：后端仍可能提交接近硬上限的分段，无法解决丢字。

方案 B：保留 32 秒为模型硬上限，新增约 29 秒安全目标，并将分段字符阈值扣除固定缓冲。
- 优点：Web、小程序和后端行为一致；即使前端未拦截，后端也会按安全阈值切段。
- 风险：分段数可能略增，但可换取生成完整性。

采用方案 B。

## 实施

- `lib/digitalHumanLimits.ts`：将分段字符上限从 `32s * safety` 调整为 `(32s * safety - padding)`，中文安全上限约从 143 字降至 139 字。
- `components/DigitalHumanModal.tsx`：拆分确认文案改为区分“32s 硬上限”和“约 29s 安全目标”。
- `digital_human_miniapp/taro/src/utils/digital-human-limits.ts` 与生成页说明同步安全目标。

## 风险与回滚

- 风险：同一长文案可能比之前多切出一段。
- 回滚：恢复分段字符上限计算为 `cps * limitSeconds * safety`，同时回退提示文案。

## 验收标准

- 约 143 个中文字符不再被当成安全单段，自动拆分阈值降到约 139 字。
- 提示不再让用户误以为可以贴着 32s 提交。
- `npm run lint` 与 `npm run typecheck` 通过。
