# 20260502 Monetization Square Config Center Plan

## 目标
将小程序“变现广场”从硬编码页面升级为后台可配置能力中心，支持运营同学在管理后台自行维护：
- 类目切换（一级菜单）
- 每个类目下的内容类型（二级卡片）
- 每个类型的演示素材（图片/视频）
- 每个类型的“做同款”能力映射（跳转/调用）
- 每个类型的默认提示词模板与变量

## 范围
- 数据层：新增 `monetization_square_configs` 配置表（JSON 配置 + 发布状态 + 版本）
- 管理后台：新增“变现广场”管理页（最小可用版）
- 服务端 API：
  - 管理端读写 API（仅管理员可访问）
  - 小程序读取 API（仅返回已发布配置）
- 小程序端：变现广场页面动态渲染配置，点击项统一路由到现有能力

## 方案对比

### 方案A：每个类目/类型拆分成关系型多表
优点：数据结构强约束、适合复杂统计分析
缺点：实现与迭代成本高，新增字段需要迁移，运营灵活性较差

### 方案B：单配置表 + JSON Schema 校验（推荐）
优点：上线快、扩展性强、支持高频迭代与灰度
缺点：需要做好服务端校验与发布规范

## 兼容性
- Next.js App Router：通过 `app/api/admin/*` 和 `app/api/miniapp/*` 承载接口
- Prisma/Supabase：新增 public schema 配置表并同步迁移
- 小程序 n8n/现有能力：不改原能力入口，仅在路由层做 action 映射

## 风险与回滚
- 风险：配置格式错误导致前端渲染异常
  - 处理：服务端写入前校验 schema，读取时兜底默认配置
- 风险：配置误发布
  - 处理：保留默认静态配置 fallback；支持快速回滚到上个版本
- 回滚策略：
  - API 返回 fallback 配置
  - 管理后台恢复上一版本 JSON（后续可增加版本历史）

## 分阶段里程碑
1. M1 数据与 API（建表、迁移、读写接口）
2. M2 管理后台页面（编辑/发布）
3. M3 小程序动态渲染 + action 路由
4. M4 联调与验收（lint/typecheck/build + 小程序构建）

## 验收标准
- 管理后台可完成“类目/类型/素材/动作/提示词”配置并保存
- 小程序“变现广场”页面按配置展示，切换类目正常
- 点击“做同款”可按 action 路由到现有能力页面
- 无配置时可回退到内置默认配置

## Tech Debt
