# 线上压力测试执行计划（Content Factory Web）

> 日期：2026-05-06
> 状态：待确认执行窗口与压测目标
> 适用环境：线上生产环境（Web / API / Supabase / n8n / 第三方模型与积分服务）

## 1. 目标

对当前线上系统进行受控压力测试，确认在真实生产链路下的容量边界、瓶颈位置与降级策略是否有效，避免无计划高并发导致线上用户受影响。

核心目标：

1. 量化 Web 页面与 API 的吞吐能力、延迟分布与错误率。
2. 验证登录态、积分扣费、任务创建、n8n 触发、Webhook 回调等关键链路在并发下的稳定性。
3. 找出数据库连接池、Supabase、n8n、第三方 API、文件上传/OSS 等外部依赖的瓶颈。
4. 产出压测报告：容量结论、风险项、扩容/限流/队列化建议、回滚与保护策略。

## 2. 重要约束与安全原则

线上压测必须满足以下前置条件，否则不执行：

- 已获得业务方/运维方明确授权。
- 已确定压测窗口，避开真实用户高峰与运营活动。
- 已准备可识别的压测账号、租户、数据前缀，避免污染真实业务数据。
- 已确认第三方服务（n8n、Supabase、模型 API、积分系统、OSS/CDN）是否允许线上压测。
- 已设置终止条件与负责人：一旦触发阈值，立即停止压测。
- 不对真实用户账号、真实余额、真实订单、真实生产素材执行破坏性操作。

建议压测流量全部带上固定标识：

- Header：`X-Load-Test: 20260506-content-factory`
- User-Agent：`content-factory-load-test/20260506`
- 测试数据命名：`loadtest_20260506_*`

## 3. 压测范围

### 3.1 第一优先级：低风险只读链路

- 首页/营销页访问。
- 登录页、公开页面访问。
- 已登录后的主应用基础页面加载。
- 只读列表接口：任务列表、素材列表、配置读取等。
- 静态资源/CDN 缓存命中情况。

### 3.2 第二优先级：核心写入链路（使用测试账号）

- 登录/会话校验。
- 创建轻量业务对象，例如测试任务、测试素材元数据。
- 查询任务状态。
- 积分预检接口（如存在）。

### 3.3 第三优先级：异步任务与外部集成链路

仅在确认预算、配额和隔离策略后执行：

- AI 内容生成任务创建。
- 图文/视频复刻任务创建。
- 分镜/数字人任务创建。
- n8n Webhook 触发。
- Webhook 回调写库。
- 积分扣费幂等性。

### 3.4 暂不建议直接压测的内容

除非另开专项并得到授权，否则不建议在生产环境做以下压测：

- 大规模真实模型生成，会产生不可控成本。
- 大文件上传/转码/视频生成，会放大 OSS、带宽、第三方模型成本。
- 支付、真实扣费、真实用户通知。
- 管理后台批量写入或删除。

## 4. 指标与通过标准

### 4.1 Web/API 指标

| 指标 | 建议阈值 |
| --- | --- |
| HTTP 5xx | < 0.5% |
| HTTP 4xx（非预期） | < 1% |
| P95 响应时间 | 只读接口 < 800ms；写接口 < 1500ms |
| P99 响应时间 | 只读接口 < 2000ms；写接口 < 4000ms |
| 首页/核心页面 TTFB | P95 < 1000ms |

### 4.2 数据库/Supabase 指标

| 指标 | 观察点 |
| --- | --- |
| DB CPU / Memory | 是否接近资源上限 |
| 连接数 | Prisma/Pooler 是否耗尽 |
| 慢查询 | 是否出现新慢 SQL |
| 锁等待 | 写入链路是否产生锁竞争 |
| RLS/鉴权错误 | 并发下是否异常升高 |

### 4.3 n8n/异步任务指标

| 指标 | 观察点 |
| --- | --- |
| Webhook 响应时间 | 是否超时或排队 |
| 工作流执行失败率 | 是否因限流/超时失败 |
| 回调成功率 | 是否重复回调、漏回调 |
| 任务积压 | 是否需要限流、队列或扩容 |
| 积分扣费 | 是否保持幂等，无重复扣费 |

## 5. 工具方案

建议使用 k6 作为主压测工具：

- 优点：脚本化、支持阶段升压、指标清晰、易接入 CI/运维流程。
- 风险：需要谨慎管理 token、测试账号、压测数据清理。

备选工具：

1. Artillery：Node 生态友好，适合 HTTP 场景。
2. Playwright：适合少量浏览器真实体验压测，不适合作为高并发主工具。

结论：以 k6 负责 API/HTTP 压测，以 Playwright/CDP 做少量 UI 真实体验验证。

## 6. 分阶段执行

### Phase 0：准备与授权

- 确认线上域名、压测窗口、联系人、终止阈值。
- 准备测试账号与测试租户。
- 备份数据库或确认可恢复点。
- 确认 Supabase / n8n / 第三方模型 / 积分服务配额与告警。
- 确认线上监控面板可用：应用日志、DB 指标、n8n 执行日志、错误追踪。

验收：准备清单全部确认后才能进入 Phase 1。

### Phase 1：基线探测（只读）

- 并发：1 → 5 → 10 VU。
- 时长：每档 3-5 分钟。
- 范围：公开页面、核心只读接口。
- 目标：确认脚本正确、监控可见、无明显错误。

验收：5xx 接近 0，P95 在预期范围内，无 DB/应用异常。

### Phase 2：阶梯升压（只读 + 轻写）

- 并发：10 → 25 → 50 → 80 VU。
- 时长：每档 5-10 分钟。
- 范围：登录态请求、任务列表、轻量测试对象创建/查询。
- 目标：识别 Web/API/DB 连接池边界。

验收：错误率与延迟未突破阈值；测试数据可清理。

### Phase 3：业务链路压测（受控写入）

- 并发：5 → 10 → 20 VU。
- 范围：任务创建、状态查询、必要的积分预检。
- 目标：验证真实用户路径的端到端稳定性。
- 注意：默认不触发高成本 AI 生成，除非使用 mock/sandbox 或低成本工作流。

验收：无重复扣费，无任务状态错乱，无数据库锁等待异常。

### Phase 4：异步链路专项（可选）

- 小流量触发 n8n 工作流。
- 检查 n8n 排队、回调、数据库更新、积分日志。
- 验证回调幂等：重复回调不重复扣费。

验收：任务闭环完整，失败可重试，成本可控。

### Phase 5：报告与修复建议

- 汇总 k6 指标、监控截图、应用日志、DB/n8n 指标。
- 输出瓶颈定位与建议：缓存、索引、连接池、限流、队列、降级、扩容。
- 清理测试数据。

验收：提交压测报告与后续修复任务列表。

## 7. 终止条件

出现任一情况立即停止压测：

- 5xx 连续 2 分钟 > 2%。
- P95 响应时间连续 3 分钟超过阈值 2 倍。
- 数据库 CPU 或连接数接近上限并持续升高。
- n8n 工作流大面积失败或积压不可恢复。
- 第三方 API 出现限流/封禁风险。
- 真实用户投诉或线上业务异常。
- 成本消耗超过预设预算。

## 8. 回滚与保护策略

- 立即停止 k6/压测进程。
- 临时开启应用层限流或 WAF 规则，屏蔽压测标识或来源 IP。
- 暂停高成本 n8n 工作流或切换到 mock/sandbox。
- 如出现数据污染，按测试数据前缀清理。
- 如出现数据库压力过高，降低应用实例并发、调整连接池或临时扩容。

## 9. 需要用户确认的信息

执行前需要确认：

1. 线上目标域名是什么？
2. 压测窗口是什么时间？预计允许持续多久？
3. 是否有固定出口 IP，是否需要加入白名单？
4. 是否允许触发 n8n 和第三方模型？预算上限是多少？
5. 是否已有监控面板地址与告警联系人？
6. 可使用哪些测试账号/租户？
7. 是否需要只做只读压测，还是包含真实任务创建？
8. 期望容量目标是多少，例如并发用户数、RPS 或日活峰值？

## 10. 交付物

- `k6` 压测脚本：只读、登录态、轻写、异步链路分开维护。
- `.env.loadtest.example`：仅包含变量名，不提交真实 token。
- 压测执行记录：时间、脚本版本、并发阶段、负责人。
- 压测报告：指标图、瓶颈分析、风险项、优化建议。
- 测试数据清理脚本或清理 SQL（需单独审核）。
