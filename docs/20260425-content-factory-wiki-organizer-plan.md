# 20260425 内容工厂原文入库 + LLM Wiki 梳理执行计划

## 目标
在现有知识库文件树中，补齐“内容工厂”核心能力：
- 提供默认内容工厂仓库（含基础目录与规则文件）。
- 支持把“原始文章/复刻解析结果”统一导入 `01-素材库/raw/`。
- 在文件树顶部提供“一键梳理”入口，将未梳理 raw 文档批量整理为 `llm-wiki`。
- 导入链路采用“非阻塞入库 + 批处理梳理”策略，提升对话流畅性。

## 备选方案对比
1. 每次导入后立即梳理（同步）
- 优点：导入后立刻可用 wiki
- 缺点：导入和对话会变慢，失败链路更复杂，成本不可控

2. 仅手动点击梳理
- 优点：最稳定、可控
- 缺点：容易忘记执行，积压原文

3. 混合策略（本次采用）
- 导入时只入 raw + 标记 `pending`
- 文件树提供“一键梳理”
- 服务端可做轻量自动批处理（小批量、非阻塞）
- 兼顾流畅性、可控性与持续产出

## 兼容性结论
- Next.js App Router：新增 API route 与前端按钮，兼容。
- Prisma/Supabase：不新增表结构，使用 `knowledge_files.metadata` 扩展状态字段，兼容。
- n8n：本次不依赖。
- LLM 调用：复用现有 `lib/cloudLLM.ts`，兼容当前云模型网关配置。

## 里程碑
1. 服务层：新增内容工厂服务（默认仓库初始化、raw 判定、wiki 梳理）。
2. API：新增文件夹梳理端点，扩展文件入库端点支持 raw/pending 元数据。
3. Chat：增强写入提示词，导入意图优先落到 `01-素材库/raw/`。
4. 复刻模块：保存到文件夹默认写 raw，并标记入库来源。
5. UI：Dashboard/Chat 文件树顶部新增“梳理”图标与 pending 计数。
6. 验证：`npm run lint`、`npm run typecheck`。

## 风险与缓解
- 风险：LLM 调用失败导致批处理中断
- 缓解：单文件隔离失败，继续处理其他文件；记录失败信息并允许重试

- 风险：wiki 文件重复/冲突
- 缓解：基于 source file id 做幂等更新，路径冲突时追加后缀

- 风险：自动批处理过频影响性能
- 缓解：限制批量大小（如每次 3-10 篇），且可手动触发为主

## 回滚策略
- 回滚新增按钮与梳理 API，不影响已有知识库读写与聊天能力。
- metadata 中新增字段为向后兼容字段，保留不影响旧逻辑。

## 验收标准
- 进入首页/对话页可看到文件树顶部“梳理”入口和待处理数。
- 导入到内容工厂 raw 后，文件 metadata 标记为 `pending`。
- 点击梳理后，成功在 `01-素材库/wiki/` 生成/更新 `llm-wiki` 文档，并更新 raw 状态。
- `npm run lint`、`npm run typecheck` 通过。
